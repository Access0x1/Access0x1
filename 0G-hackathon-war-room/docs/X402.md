## 0. Two protocol versions coexist

x402 has **v1** (original, deprecated but still published & widely deployed) and **v2** (current). Both are live on npm. Know which one your build imports.

| | v1 (legacy) | v2 (current) |
|---|---|---|
| Core npm package | `x402` (v1.2.0) | `@x402/core` (v2.10.0) |
| EVM impl | bundled in `x402` | `@x402/evm` (v2.10.0) |
| Next.js | `x402-next` | `@x402/next` |
| Express | `x402-express` | `@x402/express` |
| Payment header (client→server) | `X-PAYMENT` | `PAYMENT-SIGNATURE` |
| Requirements header (402 resp) | *(body JSON)* | `PAYMENT-REQUIRED` |
| Settlement header (200 resp) | `X-PAYMENT-RESPONSE` | `PAYMENT-RESPONSE` |
| Network format | simple name (`base-sepolia`) | CAIP-2 (`eip155:84532`) |
| Network wildcards | no (fixed list) | yes (`eip155:*`) |
| Transfer methods | EIP-3009 | EIP-3009, Permit2, ERC-7710 |
| Default validity window | 10 min (+buffer) | 1 hour |

> v1 packages carry: *"This npm package implements x402 **v1**. It is **deprecated** and will only receive **security patches**. Please migrate to **v2**."* Legacy v1 examples live at git tag `archive/legacy-v1-examples`.

Header names are the load-bearing difference. If your code reads/writes `X-PAYMENT` it is v1; if it reads/writes `PAYMENT-SIGNATURE`/`PAYMENT-REQUIRED`/`PAYMENT-RESPONSE` it is v2. (Source: `@x402/core` README "HTTP Headers" table.)

## 1. Protocol overview — the HTTP 402 flow

x402 is *"an open standard for internet native payments. It aims to support all networks (both crypto & fiat) and forms of value (stablecoins, tokens, fiat)."*

**Roles** (from root README "Terms"):
- `resource` — anything on the internet reachable over HTTP/HTTPS (webpage, file, RPC, API).
- `client` — an entity wanting to pay for a resource.
- `facilitator` — a server that verifies and executes payments for one or many networks.
- `resource server` — the HTTP server providing the API/resource.

### v2 flow (verbatim, root README "Typical x402 flow")

Steps (1)–(2) are optional if the client already knows the accepted payment details.

1. `Client` makes an HTTP request to a `resource server`.
2. `Resource server` responds with **`402 Payment Required`** and a `PaymentRequired` b64 object returned as a **`PAYMENT-REQUIRED`** header.
3. `Client` selects one of the `PaymentRequirements` and creates a `PaymentPayload` based on the `scheme` & `network` selected.
4. `Client` re-sends the request with the **`PAYMENT-SIGNATURE`** header containing the `PaymentPayload`.
5. `Resource server` verifies the payload either locally or by POSTing `PaymentPayload` + `PaymentRequirements` to the facilitator's **`/verify`** endpoint.
6. `Facilitator` verifies based on `scheme`/`network` and returns a `Verification Response`.
7. If valid, the resource server does the work. If invalid, it returns `402` + a `Payment Required Response` JSON body.
8. `Resource server` settles the payment — directly on-chain, or by POSTing `PaymentPayload` + `PaymentRequirements` to the facilitator's **`/settle`** endpoint.
9. `Facilitator` submits the payment to the blockchain based on `scheme`/`network`.
10. `Facilitator` waits for on-chain confirmation.
11. `Facilitator` returns a `Payment Execution Response` to the resource server.
12. `Resource server` returns **`200 OK`** with the resource as the body and a **`PAYMENT-RESPONSE`** header containing the base64-encoded `Settlement Response`.

> v1 differs only in header names (`X-PAYMENT` request / `X-PAYMENT-RESPONSE` response) and delivers the `PaymentRequired` object as the JSON **body** of the 402 rather than a header. (Source: legacy `x402` README "Manual Client Integration".)

### The `exact` scheme

A **scheme** is *"a logical way of moving money."* Because a scheme is logical, its implementation differs per chain, so clients and facilitators must explicitly support each `(scheme, network)` pair.

`exact` is the first scheme shipped and the default for EVM: it *"transfers a specific amount of funds from a client to a resource server"* — the server must know the exact amount in advance (e.g. pay $1 to read an article). Contrast a theoretical `upto` scheme, which transfers *up to* an amount based on resources consumed (e.g. LLM tokens). Spec: `specs/schemes/exact/scheme_exact.md` and `specs/schemes/exact/scheme_exact_evm.md`.

For `exact` on EVM: *"the Facilitator (server) pays the gas, but the Client (user) controls the exact flow of funds via cryptographic signatures... In all cases, the Facilitator cannot modify the amount or destination. They serve only as the transaction broadcaster."*

### Core v2 types (from `@x402/core` README)

```typescript
type Network = `${string}:${string}`; // e.g., "eip155:8453"

type PaymentRequirements = {
  scheme: string;
  network: Network;
  asset: string;
  amount: string;
  payTo: string;
  maxTimeoutSeconds: number;
  extra: Record<string, unknown>;
};

type PaymentPayload = {
  x402Version: number;
  resource: ResourceInfo;
  accepted: PaymentRequirements;
  payload: Record<string, unknown>;
  extensions?: Record<string, unknown>;
};

type PaymentRequired = {
  x402Version: number;
  error?: string;
  resource: ResourceInfo;
  accepts: PaymentRequirements[];
  extensions?: Record<string, unknown>;
};
```

## 2. EVM asset transfer methods — EIP-3009 & Permit2

`exact` on EVM supports three `assetTransferMethod`s (spec `scheme_exact_evm.md`). If none is specified, prioritize `eip3009` (if compatible) then `permit2`.

| Method | Use case | Notes |
|---|---|---|
| **EIP-3009** | Tokens with native `transferWithAuthorization` (e.g. USDC) | **Recommended** — simplest, truly gasless. One-time use. |
| **Permit2** | Tokens without EIP-3009 | Universal fallback for any ERC-20, via a Proxy + Permit2. |
| **ERC-7710** | Smart accounts with delegation | Paid from an ERC-7710-compatible account. One-time & multi-use. |

### 2.1 EIP-3009 `transferWithAuthorization` (gasless USDC)

The client signs an EIP-712 authorization off-chain; the facilitator submits it and pays gas. The token contract's `transferWithAuthorization` moves funds directly. USDC natively supports EIP-3009.

Standard EIP-3009 contract function (per [EIP-3009](https://eips.ethereum.org/EIPS/eip-3009)):

```solidity
function transferWithAuthorization(
    address from, address to, uint256 value,
    uint256 validAfter, uint256 validBefore, bytes32 nonce,
    uint8 v, bytes32 r, bytes32 s
) external;
```

**v2 `PaymentPayload` for EIP-3009 (verbatim from `scheme_exact_evm.md`):**

```json
{
  "x402Version": 2,
  "resource": {
    "url": "https://api.example.com/premium-data",
    "description": "Access to premium market data",
    "mimeType": "application/json"
  },
  "accepted": {
    "scheme": "exact",
    "network": "eip155:84532",
    "amount": "10000",
    "asset": "0x036CbD53842c5426634e7929541eC2318f3dCF7e",
    "payTo": "0x209693Bc6afc0C5328bA36FaF03C514EF312287C",
    "maxTimeoutSeconds": 60,
    "extra": {
      "assetTransferMethod": "eip3009",
      "name": "USDC",
      "version": "2"
    }
  },
  "payload": {
    "signature": "0x2d6a7588d6acca505cbf0d9a4a227e0c52c6c34008c8e8986a1283259764173608a2ce6496642e377d6da8dbbf5836e9bd15092f9ecab05ded3d6293af148b571c",
    "authorization": {
      "from": "0x857b06519E91e3A54538791bDbb0E22373e36b66",
      "to": "0x209693Bc6afc0C5328bA36FaF03C514EF312287C",
      "value": "10000",
      "validAfter": "1740672089",
      "validBefore": "1740672154",
      "nonce": "0xf3746613c2d920b5fdabc0856f2aeb2d4f88ee6037b8cc5d04a71a4462f13480"
    }
  }
}
```

- `payload.signature` — the 65-byte signature of the `transferWithAuthorization` operation.
- `payload.authorization` — parameters to reconstruct the signed EIP-712 message.
- `extra.name` / `extra.version` — the token's EIP-712 domain name & version (must match its domain separator). For Base Sepolia USDC these are `"USDC"` / `"2"`; for Base mainnet USDC `"USD Coin"` / `"2"` (see §5).

**Verification** (spec): signature recovers to `authorization.from`; client has sufficient balance; amount & validity window meet requirements; token & network match; simulate `transferWithAuthorization`. **Settlement**: facilitator calls `transferWithAuthorization` with the signature + authorization.

### 2.2 Permit2 (v2, any ERC-20)

For tokens lacking EIP-3009, `exact` uses Uniswap's canonical **Permit2** `permitWitnessTransferFrom` combined with a reference proxy contract **`x402ExactPermit2Proxy`** that enforces the receiver via the EIP-712 "Witness" pattern so the facilitator cannot alter the destination.

- **`x402ExactPermit2Proxy` canonical address:** `0x402085c248EeA27D92E8b30b2C58ed07f9E20001` — deployed to the same address on all supported EVM chains via `CREATE2`.
- **Canonical Permit2 address:** see [Uniswap v4 deployments](https://docs.uniswap.org/contracts/v4/deployments) (spec does not inline it). **⚠️** confirm the exact Permit2 address there.

Permit2 requires a one-time on-chain `approve(Permit2)`. Three ways to satisfy it:
- **Option A — Direct user approval:** user pays their own gas for `approve(Permit2)`.
- **Option B — Sponsored ERC-20 approval** (extension `erc20ApprovalGasSponsoring`): facilitator batches `transfer(gas)` → `ERC20.approve(Permit2)` → `settle`.
- **Option C — EIP-2612 permit** (extension `eip2612GasSponsoring`): if the token supports EIP-2612, user signs a gasless permit; facilitator calls `x402ExactPermit2Proxy.settleWithPermit()`.

If neither sponsoring extension applies and allowance is insufficient, the facilitator returns **`412 Precondition Failed`** with error code **`PERMIT2_ALLOWANCE_REQUIRED`**, signalling the client to do a one-time direct approval and retry.

Permit2 `payload` carries `signature` + `permit2Authorization` (`permitted.{token,amount}`, `from`, `spender` = the proxy address above, `nonce`, `deadline`, `witness.{to,validAfter}`). Full example and the `x402ExactPermit2Proxy` Solidity reference (with `WITNESS_TYPEHASH = keccak256("Witness(address to,uint256 validAfter)")`) are in `specs/schemes/exact/scheme_exact_evm.md`.

> Gas sponsoring applies to the **Permit2 path only** — EIP-3009 is already gasless. When both sponsoring extensions are advertised, **EIP-2612 takes priority**; the client falls back to ERC-20 approval sponsoring if the token lacks EIP-2612 (`name`/`version` absent from `extra`).

### 2.3 ERC-7710 (smart-account delegation)

Payload carries `delegationManager`, `permissionContext`, `delegator`. Verified entirely by simulating `delegationManager.redeemDelegations(...)`; settled the same way. No canonical Delegation Manager; MetaMask Delegation Framework is a notable implementation. Details in `scheme_exact_evm.md` §3.

## 3. Packages — install + minimal code

### 3.1 v1 (legacy, deprecated) — the packages many current builds still use

#### `x402` (v1 core)

```bash
npm install x402
```

Core types, schemas, and utilities used by the v1 middleware/client packages (`x402-express`, `x402-hono`, `x402-next`, `x402-fetch`, `x402-axios`). Manual v1 client integration: request the protected endpoint → server returns `402` + JSON body `{ x402Version, accepts }` → pick a requirement from `accepts` → build the payment header → retry with header on **`X-PAYMENT`** and set `Access-Control-Expose-Headers: "X-PAYMENT-RESPONSE"` to read the settlement response.

#### `x402-express` (v1 Express middleware)

```bash
npm install x402-express
```

```typescript
import express from "express";
import { paymentMiddleware, Network } from "x402-express";

const app = express();

// Configure the payment middleware
app.use(paymentMiddleware(
  "0xYourAddress",
  {
    "/protected-route": {
      price: "$0.10",
      network: "base-sepolia",
      config: {
        description: "Access to premium content",
      }
    }
  }
));

// Implement your route
app.get("/protected-route",
  (req, res) => {
    res.json({ message: "This content is behind a paywall" });
  }
);

app.listen(3000);
```

`paymentMiddleware(payTo, routes, facilitator?, paywall?)`.

#### `x402-next` (v1 Next.js middleware) — minimal integration

Page routes via middleware (`middleware.ts`):

```typescript
import { paymentMiddleware, Network } from 'x402-next';

export const middleware = paymentMiddleware(
  "0xYourAddress",
  {
    '/protected': {
      price: '$0.01',
```

```typescript
const handler = async (_: NextRequest) => {

  handler,
  "0xYourAddress",

);
```


## 4. Facilitators

A facilitator exposes **`/verify`** and **`/settle`** and handles chain interaction so sellers need no blockchain infra.

### 4.1 Testnet — `https://x402.org/facilitator` (no auth)

Used in the root/`@x402/core` examples for Base Sepolia. No API keys required. Wire it up with:

```typescript
const facilitatorClient = new HTTPFacilitatorClient({ url: "https://x402.org/facilitator" });
```

> **⚠️** The newer v2 `@x402/express` / `@x402/next` READMEs use `https://facilitator.x402.org` (host-form). Both appear in current sources; confirm which your target version expects. Treat `https://x402.org/facilitator` as the documented no-auth testnet facilitator.

### 4.2 Coinbase CDP hosted facilitator (`@coinbase/x402`)

Mainnet facilitator run by Coinbase Developer Platform. Auth via `CDP_API_KEY_ID` / `CDP_API_KEY_SECRET` (Secret API keys from [CDP Portal](https://portal.cdp.coinbase.com/projects/api-keys)). Passed automatically by the `facilitator` object from `@coinbase/x402` (which internally builds `createAuthHeaders`).

- **Networks (⚠️ secondary sources — CDP docs page 403'd):** Base, Polygon, Arbitrum, World, and Solana. ERC-20 payments via EIP-3009 (USDC, EURC) or Permit2 (any ERC-20). Polygon support launched separately per Coinbase's launch post.
- **Free tier (⚠️ secondary source):** ~1,000 transactions/month free, then ~$0.001 per transaction thereafter. Verify current pricing in CDP docs.

### 4.3 Self-hosting — `x402-rs`

The facilitator is self-hostable. **`x402-rs`** (`github.com/x402-rs/x402-rs`, crate `x402-rs`) is a Rust facilitator that verifies, settles, and monitors payments over HTTP 402. It ships a runnable binary that defaults to `http://localhost:8080` (configurable via `PORT`). Point any client/server's facilitator URL at your self-hosted instance. Other community facilitators exist (e.g. `second-state/x402-facilitator`).

## 5. Networks + USDC

x402 is network-agnostic. This build targets **Base** and **Base Sepolia**.

| Network | CAIP-2 (v2) | v1 name | Chain ID |
|---|---|---|---|
| Base Mainnet | `eip155:8453` | `base` | 8453 |
| Base Sepolia (testnet) | `eip155:84532` | `base-sepolia` | **84532** |

### USDC addresses (verbatim from `@x402/evm` `src/shared/defaultAssets.ts`)

| Network | USDC address | EIP-712 name | version | decimals |
|---|---|---|---|---|
| Base Mainnet (`eip155:8453`) | `0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913` | `USD Coin` | `2` | 6 |
| Base Sepolia (`eip155:84532`) | `0x036CbD53842c5426634e7929541eC2318f3dCF7e` | `USDC` | `2` | 6 |

- **Base Sepolia USDC `0x036CbD53842c5426634e7929541eC2318f3dCF7e`** — confirmed in the x402 source (`defaultAssets.ts` and multiple test fixtures) and in the `scheme_exact_evm.md` spec examples. This is Circle's official Base Sepolia USDC. **⚠️** cross-check on [sepolia.basescan.org](https://sepolia.basescan.org/token/0x036CbD53842c5426634e7929541eC2318f3dCF7e) (BaseScan token page was not fetched here).
- Note the EIP-712 domain **`name` differs** between chains (`USD Coin` on mainnet vs `USDC` on Sepolia). Use the exact name from the table when constructing the EIP-3009 signing domain, or take it from `extra.name` in the payment requirements.

Other default stablecoins configured in v2 (`defaultAssets.ts`), for reference: Polygon USDC `0x3c499c542cEF5E3811e1192ce70d8cC03d5c3359` (`eip155:137`), Arbitrum One USDC `0xaf88d065e77c8cC2239327C5EDb3A432268e5831` (`eip155:42161`), Arbitrum Sepolia USDC `0x75faf114eafb1BDbe2F0316DF893fd58CE46AA4d` (`eip155:421614`), plus non-EIP-3009 tokens on MegaETH / Monad / Stable / Mezo (Permit2 + EIP-2612).

### 0G chain

There is **no 0G-chain facilitator** and **no 0G entry** in x402's default stablecoin/network config. The CDP hosted facilitator supports Base, Polygon, Arbitrum, World, and Solana only (§4.2) — 0G is not among them. To use x402 on a 0G/EVM chain you would need to self-host a facilitator (§4.3) that supports that `eip155:<chainId>` and a token exposing EIP-3009 or Permit2+EIP-2612. **⚠️ UNCONFIRMED** whether any 0G token supports EIP-3009; treat 0G x402 support as not-provided out of the box.

## 6. Client side — 402 discovery, building the payment, headers

### v2 manual client flow (verbatim, `@x402/core` README)

```typescript
import { x402Client } from '@x402/core/client';
import { x402HTTPClient } from '@x402/core/http';
import { ExactEvmScheme } from '@x402/evm/exact/client';

// Create core client and register payment schemes
const coreClient = new x402Client()
  .register('eip155:*', new ExactEvmScheme(evmSigner));

// Wrap with HTTP client for header encoding/decoding
const client = new x402HTTPClient(coreClient);

// Make a request
const response = await fetch('https://api.example.com/protected');

if (response.status === 402) {
  // Extract payment requirements from response
  const paymentRequired = client.getPaymentRequiredResponse(
    (name) => response.headers.get(name),
    await response.json()
  );

  // Create and send payment
  const paymentPayload = await client.createPaymentPayload(paymentRequired);

  const paidResponse = await fetch('https://api.example.com/protected', {
    headers: client.encodePaymentSignatureHeader(paymentPayload),
  });

  // Get settlement confirmation
  const settlement = client.getPaymentSettleResponse(
    (name) => paidResponse.headers.get(name)
  );
  console.log('Transaction:', settlement.transaction);
}
```

- `getPaymentRequiredResponse(getHeader, body)` — reads the `PAYMENT-REQUIRED` header (or 402 body) into a `PaymentRequired`.
- `createPaymentPayload(paymentRequired)` — selects an `accepts` entry and signs (EIP-3009 by default for EVM).
- `encodePaymentSignatureHeader(payload)` — returns `{ "PAYMENT-SIGNATURE": "<base64>" }` to attach to the retry.
- `getPaymentSettleResponse(getHeader)` — decodes the `PAYMENT-RESPONSE` header; `.transaction` is the settlement tx hash.

### Header cheat-sheet

| Direction | v2 header | v1 header | Encoding |
|---|---|---|---|
| 402 response → client (requirements) | `PAYMENT-REQUIRED` | *(402 JSON body `{x402Version, accepts}`)* | base64 JSON |
| client retry → server (signed payload) | `PAYMENT-SIGNATURE` | `X-PAYMENT` | base64 JSON |
| 200 response → client (settlement) | `PAYMENT-RESPONSE` | `X-PAYMENT-RESPONSE` | base64 JSON |

For v1 browser clients, also set `Access-Control-Expose-Headers: "X-PAYMENT-RESPONSE"` so the settlement header is readable cross-origin. In practice most clients never build headers by hand — use `@x402/fetch` / `@x402/axios` (v2) or `x402-fetch` / `x402-axios` (v1), which wrap `fetch`/Axios and run the whole 402 → sign → retry loop automatically.

### Client policies & hooks (v2)

`x402Client.fromConfig({ schemes, policies })` lets you register per-network schemes and add `policies` — e.g. reject requirements above a max price:

```typescript
policies: [
  (version, reqs) => reqs.filter(r => BigInt(r.amount) < BigInt('1000000')),
]
```

Lifecycle hooks (`onBeforePaymentCreation`, `onAfterPaymentCreation`, `onPaymentCreationFailure` on the client; `onBeforeVerify`/`onAfterVerify`/`onBeforeSettle`/`onAfterSettle` on server & facilitator) allow aborting/recovering/logging around payment creation and settlement.
